# Server

Source under `server/`. FastAPI + `python-multipart` + PyAV + OpenCV.
`uv`-managed venv at `server/.venv` (Python 3.13).

## Key Python modules

- `schemas.py` — Pydantic types (`IntrinsicsPayload`, `BlobCandidate`, `FramePayload`, `PitchPayload`, `CalibrationSnapshot`, `TriangulatedPoint`, `SessionResult`, `DeviceIntrinsics`, `MarkerRecord` + draft/batch variants, `SyncReport` / `SyncResult` / `SyncRun` / `SyncLogEntry`) plus the `DetectionPath` / `TrackingExposureCapMode` enums and in-memory `Device` / `Session` / `StoredPitch` dataclasses. `BallisticSummary` and `CaptureMode` are gone (RANSAC fit removed). `SegmentRecord` carries the persisted shape of `segmenter.Segment` — `(p0, v0, t_anchor, t_start, t_end, indices, original_indices, rmse_m, speed_kph)`. `SessionResult.segments_by_path` is the full per-path fit surface (`live` / `server_post` each get their own segment list); legacy `SessionResult.segments` follows the same explicit legacy path as `SessionResult.points` and never borrows live when server_post was requested but missing. The segmenter runs at result build / recompute time inside `session_results.stamp_segments_on_result` (which pre-sorts each path's `triangulated_by_path[path]` by `t_rel_s` so `original_indices` index into a time-sorted list); the viewer PATH switch then flips between those persisted per-path segments without recomputing on the client. Sample curves are NOT persisted — clients reconstruct via `p0 + v0·τ + ½·G·τ²`. Legacy result JSONs that carried retired `ballistic_*` / `peak_z_m` / `ballistic_speed_mph` fields are historical artifacts; one-shot migration scripts were removed with the experimental-phase migration shims. Current code treats result JSONs as a materialized cache: startup reloads valid fresh `data/results/session_<sid>.json` directly, and only rebuilds/stamps modern segment surfaces from pitch payloads when the result file is missing, stale relative to its pitch JSONs, corrupt, or explicitly recomputed. The standalone `/fit/{sid}` page is retired. `PitchPayload` carries two parallel frame buckets — `frames_live`, `frames_server_post` — each independently populated by the matching path; `PitchPayload.server_post_ran_at` (optional unix timestamp) is stamped at completion of a successful detection run for the viewer's "last run" stamp. `FramePayload.candidates: list[BlobCandidate]` is the Phase B wire shape: iOS ships every HSV-and-shape-gate-passing blob per frame (no top-K cap, no winner pre-selection); the server fans them all out through `pairing.triangulate_cycle`, which emits every geometrically-plausible pair (gated only by the absolute emit ceiling — see `pairing.py` entry above). **Dict-canonical detection-config storage.** Both `PitchPayload` and `SessionResult` persist detection-time config in algorithm-keyed dicts: `frames_by_algorithm: dict[str, list[FramePayload]]` (only on `PitchPayload`) and `config_used_by_algorithm: dict[str, DetectionConfigSnapshotPayload]` (both). The dict key is an `algorithm_id` — `ios_capture_time` for the live path, runnable detector ids (`v11_hsv_cc`, `hybrid_28d`, …) for server_post. `DetectionConfigSnapshotPayload` (`schemas.py:174`) is the value model (algorithm_id + params + optional `preset_name`). Legacy flat fields (`frames_live`, `frames_server_post`, `live_config_used`, `server_post_config_used`, `hsv_range_used`, `shape_gate_used`) are **computed-field projections** of the dict, NOT stored: `_PITCH_PERSIST_EXCLUDE` and `_RESULT_PERSIST_EXCLUDE` strip them from `.model_dump(mode="json")` before disk write, so the dict is the only on-disk truth. The `server_post` projection (`server_post_config_used` / `frames_server_post`) reads through an `active_server_post_algorithm_id` pointer — when an operator re-runs a different algorithm the pointer flips and projections follow, but prior algorithm buckets stay in the dict (multi-algorithm history at the storage layer). On `state.record()` partial merges, the merged payload's `active_server_post_algorithm_id` is preserved across incomplete writes that lack a fresh pointer.

`SessionResult.gap_threshold_m` is the only operator-stamped per-session pairing tunable that survives. The legacy `SessionResult.cost_threshold` has been removed (b15a611 + bc0e92e) — cost gating is now **algorithm-owned**, declared per-entry in `server/algorithms/__init__.py` via `cost_threshold_for_algorithm(algorithm_id)` and applied server-side in pairing before segmentation. `POST /sessions/{sid}/recompute` body now accepts only `{gap_threshold_m: float}`; Cost has no operator-tunable analogue. Fresh result builds seed `SessionResult.gap_threshold_m` from the currently saved global pairing tuning; the viewer renders the full emitted set and the Gap slider masks client-side, so dragging is purely a render-time mask and Apply re-runs the segmenter on the stamped-gap subset.

`BlobCandidate` still carries `(px, py, area, area_score, aspect, fill, cost)` — `aspect`/`fill` are server-populated by detection.py and consumed by the shape-prior cost stamped onto each candidate. `static/viewer/50_canvas.js::_passCostFilterPoint` is now a no-op stub (`return true`) kept only so call sites compile; client-side cost filtering is retired. `TriangulatedPoint.cost_a` / `cost_b` are still on the wire (they ride along for any future debugging visualisation) but no longer fed into a render-time predicate.

**Freezing semantics.** `routes/pitch.py` stamps the live session's frozen snapshot under `config_used_by_algorithm[IOS_CAPTURE_TIME_ALGORITHM_ID]` on `/pitch` upload; the `/sessions/{sid}/runs/{algorithm_id}` handler stamps the exact algorithm + config snapshot it ran with under `config_used_by_algorithm[algorithm_id]` on completion and flips `active_server_post_algorithm_id` to that id. `state.ingest_live_frame` freezes `live.pairing_tuning` + the live-bucket snapshot on first ingest only (dashboard slider edits during an active session land on the next session). `state.record()` preserves the algorithm-bucket dicts across partial merges, so a follow-up write that only carries live frames or a timestamp cannot silently erase the session's frozen detection identity. Selector cost weights are not part of this snapshot — they're locked at `_W_ASPECT=0.6` / `_W_FILL=0.4` module constants in `candidate_selector.py`. `reprocess_sessions.py` defaults to current `data/*.json` so the operator's tuning workflow (edit → rerun → see if it improves) actually picks up changes; opt-in `--use-frozen-snapshot` rebuilds detection from the `config_used_by_algorithm` projection (falls back to disk on legacy pitches with a logged warning). The frozen stamps on disk are always re-written at end of run with whatever config the run actually used, so the "what config produced this pitch" question stays answerable even after a default-mode rerun overwrites them. `CalibrationSnapshot` is the body of `POST /calibration`. `camera_id` is `^[A-Za-z0-9_-]{1,16}$`; `session_id` regex is `^s_[0-9a-f]{4,32}$` (server mints `s_` + 8 hex chars via `secrets.token_hex(4)`; 32-char upper bound is forward-compat slack). Both path-safe.
- `detection.py` — OpenCV-backed HSV blob extractor + multi-candidate scorer used by the `v11_hsv_cc` algorithm. `HSVRange.default()` is `25,55,90,255,90,255` (fluorescent yellow-green tennis ball — fallback for headless tests / first boot); the dashboard persists the v11 live config (HSV + shape_gate pair) atomically in `data/detection_config.json`. The dashboard's Apply UX is **algorithm-aware** (d963e4a): the form pulls the algorithm picker + params schema from `GET /algorithms`, dynamically generates the params editor for the chosen algorithm, and POSTs to `POST /presets` with a generic body `{name, label, algorithm_id, params}`. For a `v11_hsv_cc` preset that becomes the live target, the server writes the preset file and atomically switches the live `DetectionConfig.preset` to it (`state.set_detection_config(...)` under the lock + WS settings broadcast); for non-v11 presets the file is saved on disk only (live is untouched, WS is silent — non-v11 presets are server_post-only). Pure switches without writing a new file go through `POST /presets/active {name, target}`. Slider drag is local UI only; preset filenames are immutable by name (saving under an existing slug returns 409). The legacy per-section endpoints (`/detection/hsv`, `/detection/shape_gate`, `/detection/candidate_selector`) are retired — calls return 404. On boot, `detection_config.load_or_migrate` migrates legacy `data/{hsv_range,shape_gate,candidate_selector_tuning}.json` into the unified file once and deletes them. Subsequent boots read only the new file. The named preset library is **disk-backed**: `presets.py` exposes `load_preset` / `list_presets` / `save_preset` / `delete_preset` over `data/presets/<name>.json`, with `seed_builtins` writing `tennis`, `blue_ball`, and `hybrid_28d_blue_ball` seeds on first boot if their files do not exist. Built-ins survive operator edits (the seed step never overwrites an existing file) and an `rm` + restart restores the canonical seed. Operator-facing CRUD: `GET /presets` (list), `GET /presets/{name}` (single), `POST /presets` body `{name, label, algorithm_id, params}` (409 on duplicate; on success atomically switches active to the newly-saved name when it's a v11 preset and broadcasts WS settings), `POST /presets/active` body `{name: str, target: "live" | "server_post"}` — pure switch by name. Error matrix: 400 if `name` or `target` field is missing; 404 if preset is unknown; 422 if `target` value is not one of `{"live", "server_post"}`; 422 if `target=live` and `preset.algorithm_id ≠ v11_hsv_cc` (live path is v11_hsv_cc-only). On success: `target=live` writes `data/detection_config.json` and broadcasts a `settings` WS push to all connected iOS devices; `target=server_post` writes `data/active_server_post_preset.json` and does **not** push anything over WS, `DELETE /presets/{name}` (409 if the slug is currently active on either slot — operator must switch active first; 404 if missing). `PUT /presets/{name}` retired — preset filenames are immutable. Slug is restricted to `[a-z0-9_]{1,32}` so it doubles as a filename safely; the operator-facing `label` is free-form and HTML-escaped at render. Identity-claim validation on `POST /detection/config` reads from disk, so a freshly-saved v11 preset becomes claimable as `preset=<name>` immediately without restart. Hue is OpenCV's 8-bit convention (0-179, standard 0-360° halved). `detect_ball(bgr, hsv_range, ...)` runs `cv2.inRange` + `cv2.connectedComponentsWithStats`, applies the operator-tunable `ShapeGate` (aspect / fill thresholds, persisted inside the unified detection_config), and feeds the surviving blobs to `candidate_selector.score_candidates`. The selector cost is **track-independent and scale-invariant**: `cost = _W_ASPECT·aspect_pen + _W_FILL·fill_pen` where `aspect_pen` is `(1 - aspect)` normalised to a fixed floor (0.5), and `fill_pen` is `|fill - 0.68|/0.68` (0.68 = empirical median for the project ball). There is no size term — absolute area in pixels is a function of distance (a 240 fps pitch sweeps r ≈ 4 → 25 px), so it cannot be a property of "ball-likeness." Area still gates entry via `MIN_AREA = 20`, but is not a cost input. Weights are locked at `_W_ASPECT=0.6 / _W_FILL=0.4` module constants in `candidate_selector.py` — there is no runtime tunable, no disk file, no dashboard card. The same selector runs in the live path via `live_pairing._resolve_candidates`; iOS ships aspect/fill on every candidate (lockstep wire contract). `BlobCandidate.aspect` / `.fill` are now **required** wire fields (no Optional / no fallback to 0); a missing value fails Pydantic validation rather than silently zero-penalty. **No `chain_filter` step** — `segmenter.find_segments` (run at SessionResult build time, persisted on `SessionResult.segments`) supersedes its flicker/jump heuristics with a residual + running-RMSE gate against the multi-segment ballistic LSQ.
- `algorithms/__init__.py` — **algorithm registry**. Module-level dict of `AlgorithmEntry` records keyed by `algorithm_id`. Current entries: `v11_hsv_cc` (runnable; HSV + connectedComponents + shape gate, driven by `detection.py`), `hybrid_28d` (runnable; the 28-dimensional hybrid detector), `ios_capture_time` (NOT runnable — sentinel for the live-bucket dict key; `IOS_CAPTURE_TIME_ALGORITHM_ID` constant). Each runnable entry carries: `algorithm_id`, human label, params JSON schema (consumed by the dashboard Apply form generator and by `POST /sessions/{sid}/runs/{algorithm_id} {params}` validation), `cost_threshold` (algorithm-owned pairing cost gate — see `cost_threshold_for_algorithm(algorithm_id)`), and the run function. `run_detection(algorithm_id, mov_path, video_start_pts_s, params, …)` is the unified entry point — routes detection through the chosen algorithm without callers caring about the underlying module. `validate_id(slug)` accepts any registered id (including non-runnable `ios_capture_time`) — used wherever the id is a dict key. `validate_runnable_id(slug)` rejects non-runnable ids with 422 — used by `/sessions/{sid}/runs/{algorithm_id}` and `POST /presets`. Boot-time drift guards `_check_schemas_constant_drift` / `_check_legacy_bucket_in_registry` raise loudly on import if `schemas.py` constants or persisted JSONs reference an unregistered algorithm id; this catches a stale data dir or a half-deleted entry before serving traffic.
- `routes/algorithms.py` — `GET /algorithms` returns a sorted list of registered runnable entries (algorithm_id, label, params schema, cost_threshold); `GET /algorithms/{algorithm_id}` returns the single entry (404 on unknown ids INCLUDING non-runnable `ios_capture_time`, so dashboard form generators can rely on it). The dashboard's preset Apply form fetches `/algorithms` once on open and `/algorithms/{algorithm_id}` per algorithm switch to rebuild the params editor.
- `video.py` — PyAV decoder. `iter_frames(mov_path, video_start_pts_s) → Iterator[(absolute_pts_s, bgr)]`. Reconstructs absolute session-clock PTS: `absolute_pts_s = video_start_pts_s + (frame.pts − first_pts) * time_base`, so every decoded frame's timestamp sits on the same clock as `sync_anchor_timestamp_s`. `count_frames(path)` is a cheap second-pass counter used by tests.
- `pipeline.py` — glue. `detect_pitch(mov_path, video_start_pts_s, hsv_range?, frame_iter?, *, should_cancel?, shape_gate?, progress?)` decodes the MOV and runs the byte-aligned HSV → CC → shape gate → shape-prior selector pipeline (no MOG2, no morphology, no `expected_radius_px`, no chain_filter, no temporal-prior plumbing). Synthesises one `FramePayload` per decoded sample on the iOS session clock. Tests inject a stub iterator to avoid real decoding.
- `pairing.py` — `triangulate_cycle(a, b)` does A/B frame pairing within an 8 ms window of anchor-relative time (`timestamp_s − sync_anchor_timestamp_s`) and runs ray-midpoint triangulation. Each matched frame iterates `cands_a × cands_b` (cartesian fan-out). **Pairing emits the FULL set** of geometrically-plausible candidate pairs, gated only by absolute module-level ceilings `_EMIT_COST_CEILING=5.0` / `_EMIT_GAP_CEILING_M=5.0` — disk/memory protection, NOT operator-tunable. `tuning.gap_threshold_m` is the only OPERATOR per-session pairing filter that the segmenter (`session_results.stamp_segments_on_result`) and the viewer's slider (`static/viewer/50_canvas.js::_passResidualFilter`) apply DOWNSTREAM; it does NOT gate emit. **Cost gating is algorithm-owned** (`algorithms.cost_threshold_for_algorithm`) — applied server-side before segmentation; the legacy operator-stamped `tuning.cost_threshold` and viewer `_passCostFilterPoint` are retired (`_passCostFilterPoint` survives as a no-op stub that always returns `true`). Every emitted `TriangulatedPoint` still carries `cost_a` / `cost_b` (source candidates' selector cost) for offline debugging; `triangulated_by_path` on disk is the full emitted set so reload + Apply can re-fit any subset without re-pairing. `_ray_for_frame` prefers the pixel path whenever `px`/`py` are present (server detection always produces them); zero-distortion is the default when `intrinsics.distortion` is absent, which is numerically equivalent to the legacy angle ray. Requires intrinsics + homography on both cameras.
- `triangulate.py` — `recover_extrinsics` decomposes `H = K [r1 r2 t]` (Zhang's planar method), orthonormalizes via SVD, flips sign if `t[2] < 0`. `triangulate_rays` solves the 2×2 system for the shortest segment between two 3D rays. `undistorted_ray_cam` applies OpenCV 5-coefficient undistortion.
- `reconstruct.py` — pure-geometry scene builder used by the viewer endpoints. `build_scene(session_id, pitches, triangulated)` returns a `Scene` with cameras, rays (tagged with `source` = `live` / `server` so the viewer can split them), ground traces per path (`ground_traces`, `ground_traces_live`), and triangulated points per path (`triangulated`). Viewer JS keeps the two pipelines independent via these per-path collections; they are never merged. `build_calibration_scene(calibrations)` builds the dashboard-canvas scene without rays. Both paths share `_camera_view_from_intrinsics_and_homography` so poses are byte-for-byte identical across `CalibrationSnapshot` and post-triangulation `PitchPayload`. **`t_rel_s` on rays and ground traces MUST share the same clock as `frames[*].t_rel_s` produced by `routes/viewer.py::_videos_for_session`** — both fall back to `video_start_pts_s` when `sync_anchor_timestamp_s` is None. A stale `or 0.0` fallback here would emit absolute PTS (~14681 s) while frames and the video element stay at anchor-relative (0–5 s), breaking viewer playback's scrubber→ray lookup.
- `chirp.py` — `chirp_wav_bytes()` builds the reference sync chirp WAV (2→8 kHz linear sweep, 100 ms, Hann-windowed, surrounded by 0.5 s silence) and is `@functools.lru_cache(maxsize=1)`'d so `GET /chirp.wav` reuses the exact bytes across requests.
- `scene_runtime.py` / `render_dashboard_page.py` / `viewer_page.py` (+ split helpers `render_dashboard_{client,devices,events,intrinsics,session,style}.py`) — Three.js dashboard + viewer HTML. Same PHYSICS_LAB palette. The shared 3D scene runtime lives client-side at `static/threejs/scene_runtime.js` (renderer + camera + OrbitControls + the static layers ground / plate / strike zone / world axes); per-page layer modules in `static/threejs/{dashboard,viewer}_layers.js` add dynamic content; `static/threejs/fit_curves_layer.js` is the shared fit-segment module — Line2 fat lines (vendored under `static/threejs/vendor/lines/`) so the operator can dial linewidth in screen-px, dashed pre/post extension, raycaster hover tooltip showing instantaneous km/h via the closed-form derivative `v(t) = v0 + g·(t − t_anchor)`. `static/threejs/playback_marker.js` is the shared current-ball policy: playback pose is always evaluated from the active fitted segment; raw triangulated points are observations and never drive the current-ball marker. `static/threejs/ball_marker.js` renders that marker as one procedural 3D baseball mesh shared by dashboard and viewer: an off-white sphere plus explicit front-hemisphere seam/stitch line geometry, not a UV texture, so the seams remain readable at tiny marker sizes. Layer chips on both surfaces (`Traj` / `Fit`) are popovers — the chevron expands a panel with per-layer config sliders (`PT` for traj points, `LW` + `EXT` for fit linewidth and dashed extension), wired by `bindLayerPopovers` from the shared module. Strike zone is now server-owned geometry rather than a hard-coded JS box: `scene_runtime.py::scene_theme()` embeds the current `strike_zone` payload (front face, back face, connectors, front-face 3×3 grid, plus bounds), and the runtime exposes `setStrikeZone(...)` so the dashboard can redraw after a settings POST without a page reload. `viewer_layers.js` now treats PATH as the viewer's full data-source selector: `live` vs `server_post` switches 3D trajectory, fit curves, active fit marker, speed badge, rays, ground traces, and 2D BLOBS together. Dashboard 3D has a simpler bottom-of-canvas marker-only playback transport for the selected session/path and deliberately does not sync video. Dashboard and viewer playback-rate selections persist via browser `localStorage`; server-owned session / detection / sync tunables remain in disk-backed server stores. Colors still encode scene role (segment palette for points + fit, stable per-camera hues for rays / ground) rather than path. Three.js is self-hosted in `static/threejs/vendor/`; `main.py` mounts `/static/` via `StaticFiles` and fails loud at boot if the vendored files are missing (`vendor_files_present`).
- `main.py` — FastAPI app + `State` class (thread-locked). WS handler `ws_device` lives in `routes/device_ws.py`. `State.pitches` keyed by `(camera_id, session_id)`; when both A and B for a session arrive, per-path triangulation runs (see `session_results.py`) and the `SessionResult` is cached in `State.results[session_id]` plus atomically written to `data/results/session_<sid>.json`. Startup reads pitch JSONs first, then restores each result from disk if the result validates, matches the session id, and is newer than every pitch JSON for that session; otherwise it rebuilds and refreshes only that one result file. The `/pitch` handler lives in `routes/pitch.py`: multipart `payload` is always required, `video` arrives on every real upload (iOS records unconditionally post-PR61) but the handler accepts frame-only payloads so tests stay slim. `/pitch` no longer kicks off server-side detection; it just archives the MOV. Operator-triggered detection runs come in via two endpoints that share the same `_dispatch_server_post` tail (snapshot → BackgroundTask). Primary surface is `POST /sessions/{sid}/runs/{algorithm_id}` (Phase 7): URL pins the algorithm, body carries either `{preset_name: str}` (load preset, validate `preset.algorithm_id == URL`) or `{params: dict}` (ad-hoc one-off; params are validated against the detector's `params_schema` and the resulting snapshot's `preset_name` is null). Error matrix: 400 if algorithm slug malformed, 404 if algorithm or preset unknown, 422 if algorithm is a non-runnable data source (`ios_capture_time`) / both body fields present / neither body field present / preset.algorithm_id mismatches URL / params fail schema. The deprecation-alias `POST /sessions/{sid}/run_server_post` is kept for HTML form callers (events-row "Run srv" + viewer "Rerun server" embed `<select name="preset_name">` from `GET /presets`); body is preset-name-only, the snapshot's algorithm_id is derived from the preset, behaviour after dispatch is identical. The candidate gate is "MOV on disk + session not trashed" (re-runs on already-completed sessions are allowed). The captured snapshot is threaded into the BackgroundTask so a concurrent dashboard slider edit can't contaminate an already-queued reprocess. The resulting `SessionResult.config_used_by_algorithm[algorithm_id]` is what events / viewer render via the active-pointer projection (`server_post_config_used`); rerunning the same algorithm overwrites that bucket, running a different algorithm leaves prior buckets in place (multi-algorithm history preserved at the dict layer). Live `frame` / `path_completed` messages on `WS /ws/device/{cam}` go through `state.ingest_live_frame` + `state.live_rays_for_frame` (one ray per shape-gate-passing candidate, fan-out parity with the post-pitch viewer scene); `state.persist_live_frames` flushes the WS buffer onto the pitch JSON so a restart can replay. Runtime settings now own `strike_zone.batter_height_cm` and `sync_params`; `State.strike_zone()` derives the current 3D geometry from that one height value using the ABS-style proxy `z_bottom_m = 0.270H`, `z_top_m = 0.535H` (`H = batter_height_cm / 100`), while width stays at the 17-inch plate width and the 3D scene keeps full plate depth for readability. `/status` surfaces the derived bounds/geometry for dashboard refresh. Other endpoints live under `routes/{sessions,calibration,markers,settings,sync,camera,viewer,pitch,presets}.py` — grep `@router\.(get|post|delete)` there for the canonical list. Notable groups: `/detection/config` (low-level pair setter — used by tests and direct curl; the dashboard UI no longer reaches it. `POST /detection/config/reset_to_preset` retired; switch active via `POST /presets/active`. Legacy per-section `/detection/{hsv,shape_gate,candidate_selector}` endpoints are retired and return 404), `/settings/{chirp_threshold,mutual_sync_threshold,heartbeat_interval,tracking_exposure_cap,capture_height,sync_params,strike_zone}`, `/sessions/{sid}/runs/{algorithm_id}` (primary surface — preset_name XOR params; full error matrix above) + `/sessions/{sid}/run_server_post` (deprecation alias for the preset-name path) + `/sessions/{sid}/active_run` (pure pointer flip — no detection; switches active server_post bucket to an already-run algorithm_id, broadcasts `fit` SSE with `cause: "active_run_switch"`; see protocols.md for full contract) + `/sessions/{sid}/cancel_processing` (operator-triggered server-post detection + Cancel button on the events row), `/sessions/{sid}/star` + `/sessions/{sid}/unstar` (events-row star toggle; persisted on `SessionResult`), `/sessions/{sid}/trash` + `/sessions/{sid}/restore` + `/sessions/{sid}/delete` (three-step retention: trash moves session to trashed state, restore un-trashes, delete permanently removes MOV + pitch + result files), `/presets` CRUD: `GET /presets` (list), `GET /presets/{name}`, `POST /presets` (create + atomic active switch; 409 on duplicate slug), `POST /presets/active` body `{name, target}` (pure switch; 400 on missing field, 404 on unknown preset, 422 on bad target or v11 mismatch; live target → WS broadcast, server_post target → no WS), `DELETE /presets/{name}` (409 if currently active), `PUT /presets/{name}` retired (preset filenames are immutable — save under a new slug instead), `/sync/*` (mutual sync coordinator), `/calibration/intrinsics/*` (per-device ChArUco upload). The standalone `/fit/{sid}` page is retired (`routes/fit.py` + `render_fit.py` deleted; viewer PATH now switches among the persisted `SessionResult.segments_by_path` surfaces). Removed: `/sessions/set_mode`, `/detection/paths` (legacy path-selection UI gone), `/detection/chain_filter` (chain_filter stack removed; segmenter supersedes).
- `state_events.py` / `session_results.py` / `detection_paths.py` / `state_detection.py` / `status_view.py` / `ws_messages.py` (extracted from `state.py`; PR #117 / #119 carved out the first three, the state-py-split branch carved out the last three) — `state_events.py` builds the `/events` dashboard rows (per-session, per-pipeline counts + path status pills); `session_results.py::rebuild_result_for_session` is the authoritative constructor for `SessionResult`, coordinating per-path triangulation; `detection_paths.py` resolves which frame bucket (`frames_live` / `frames_server_post`) is authoritative for a given `(pitch, path)` pair; `state_detection.py` holds the live / server_post detection-state mutations (live frame ingest, frozen-config stamping, active-algorithm pointer flips); `status_view.py` holds dashboard read-only projections (`summary`, `latest`, `auto_cal_status`, calibration last-solve summaries, live session summary); `ws_messages.py` holds the device-WS command derivation (`commands_for_devices` + `_DISARM_ECHO_S`). Auto-cal reruns preserve existing intrinsics and only update homography (Phase-3 contract).
- `calibration_auto.py` — pure auto-calibration business logic extracted from `routes/calibration.py` (PR `a9d46f6`, brought `routes/calibration.py` down from 1162 → 298 lines). Owns the JPEG → 4:3-to-16:9 center-crop → ArUco detect → intrinsics derivation (ChArUco prior / cached snapshot / `_IPHONE_MAIN_CAM_HFOV_RAD = 1.2885` fallback at `calibration_auto.py:51`) → planar homography → PnP refinement → reprojection gate → rebuild in canonical 1920×1080 video basis. `_run_auto_calibration` is the single entrypoint; FastAPI route handlers in `routes/calibration.py` are thin wrappers. Module raises `HTTPException` directly so callers don't need try/except wrapping.
- `live_pairing.py` — `LivePairingSession` buffers WS-streamed frame messages and pairs A/B within the 8 ms window on the fly. iOS ships every shape-gate-passing candidate per frame as `{px, py, area, area_score, aspect, fill}` (the on-wire candidate shape post-abfa422; iOS hand-encodes via `LiveFrameDispatcher.swift:93-95` so aspect/fill must be added explicitly there alongside the Codable struct). `area_score` is **wire-carried but not a selector-cost input** — kept only as a viewer BLOBS-overlay sort fallback (see `BlobCandidate` docstring in `schemas.py`). `cost` is **not** sent by iOS; the server stamps it onto every candidate inside `_resolve_candidates` after running the shape-prior selector, so persisted JSONs and downstream consumers see a populated `cost` field. `_resolve_candidates` runs the shared shape-prior `candidate_selector` (frame-local — no temporal state, no `area_score` term in the cost). The selector cost is `_W_ASPECT·aspect_pen + _W_FILL·fill_pen` with weights locked as module constants (no runtime tunable, no disk file); absolute `area` still gates entry via `MIN_AREA = 20` but is not a cost input (a 240 fps pitch sweeps r ≈ 4 → 25 px so absolute size cannot be a property of "ball-likeness"). Triangulated points are emitted as soon as the second cam's matching frame lands; the server broadcasts `path_completed` / triangulated-point events on the same WS channel so the dashboard can draw before the session ends.
