# Server

Source under `server/`. FastAPI + `python-multipart` + PyAV + OpenCV.
`uv`-managed venv at `server/.venv` (Python 3.13).

## Key Python modules

- `schemas.py` — Pydantic types (`IntrinsicsPayload`, `BlobCandidate`, `FramePayload`, `PitchPayload`, `CalibrationSnapshot`, `TriangulatedPoint`, `SessionResult`, `DeviceIntrinsics`, `MarkerRecord` + draft/batch variants, `SyncReport` / `SyncResult` / `SyncRun` / `SyncLogEntry`) plus the `DetectionPath` / `TrackingExposureCapMode` enums and in-memory `Device` / `Session` / `StoredPitch` dataclasses. `BallisticSummary` and `CaptureMode` are gone (RANSAC fit removed; segmenter on `/fit/{sid}` page is now the only ballistic surface). `PitchPayload` carries two parallel frame buckets — `frames_live`, `frames_server_post` — each independently populated by the matching path; `PitchPayload.server_post_ran_at` (optional unix timestamp) is stamped at completion of a successful detection run for the viewer's "last run" stamp. `FramePayload.candidates: list[BlobCandidate]` is the Phase B wire shape: iOS ships every HSV-and-shape-gate-passing blob per frame (no top-K cap, no winner pre-selection); the server fans them all out through `pairing.triangulate_cycle` and lets the skew-line residual gate (sole authority — `PairingTuning.gap_threshold_m`, per-session via the viewer's Gap slider) plus the segmenter's ballistic fit pick the real trajectory; downstream stages (segmenter, viewer) trust pairing's emit set and do not re-filter. `BlobCandidate` carries `(px, py, area, area_score, aspect, fill, cost)` — `aspect`/`fill` are server-populated by detection.py and consumed by the shape-prior cost stamped onto each candidate so the viewer's Cost slider (sibling of the Gap slider in the per-session tuning strip) can filter cheap distractors. `SessionResult.cost_threshold` and `SessionResult.gap_threshold_m` persist the operator's chosen per-session values; both are sent in the body of `POST /sessions/{sid}/recompute` and re-stamped on the resulting result so the viewer header re-initialises after reload. `PitchPayload.{hsv_range_used, shape_gate_used, candidate_selector_tuning_used}` and the matching `SessionResult.*_used` mirrors freeze the **detection-time config** at first persist of each pitch — `routes/pitch.py` stamps from the live session's first-ingest snapshot on /pitch upload and re-stamps from the actual server_post call args on completion; `state.ingest_live_frame` freezes `live.tuning` / `live.pairing_tuning` / `live.hsv_range_used` / `live.shape_gate_used` on first ingest only (dashboard slider edits during an active session land on the next session). `reprocess_sessions.py` defaults to current `data/*.json` so the operator's tuning workflow (edit slider → rerun → see if it improves) actually picks up changes; opt-in `--use-frozen-snapshot` reuses `pitch.*_used` for reproducibility audits and falls back to disk on legacy pitches with a logged warning per missing field. The frozen stamps on disk are always re-written at end of run with whatever config the run actually used, so the "what config produced this pitch" question stays answerable even after a default-mode rerun overwrites them. Same intent as cd87995's PairingTuning-on-SessionResult fix, generalised to detection. `CalibrationSnapshot` is the body of `POST /calibration`. `camera_id` is `^[A-Za-z0-9_-]{1,16}$`; `session_id` regex is `^s_[0-9a-f]{4,32}$` (server mints `s_` + 8 hex chars via `secrets.token_hex(4)`; 32-char upper bound is forward-compat slack). Both path-safe.
- `detection.py` — OpenCV-backed HSV blob extractor + multi-candidate scorer. `HSVRange.default()` is `25,55,90,255,90,255` (fluorescent yellow-green tennis ball — fallback for headless tests / first boot); the dashboard persists the live range in `data/hsv_range.json` via `POST /detection/hsv` (presets `tennis` / `blue_ball`, or six explicit ints). `HSVRange.from_env()` still reads `BALL_TRACKER_HSV_RANGE="hMin,hMax,sMin,sMax,vMin,vMax"` as a last-resort override for headless test runs. Hue is OpenCV's 8-bit convention (0-179, standard 0-360° halved). `detect_ball(bgr, hsv_range, ...)` runs `cv2.inRange` + `cv2.connectedComponentsWithStats`, applies the operator-tunable `ShapeGate` (aspect / fill thresholds, `data/shape_gate.json`), and feeds the surviving blobs to `candidate_selector.select_best_candidate`. The selector cost is **track-independent and scale-invariant**: `cost = w_aspect·aspect_pen + w_fill·fill_pen` where `aspect_pen` is `(1 - aspect)` normalised to a fixed floor (0.5), and `fill_pen` is `|fill - 0.68|/0.68` (0.68 = empirical median for the project ball). There is no size term — absolute area in pixels is a function of distance (a 240 fps pitch sweeps r ≈ 4 → 25 px), so it cannot be a property of "ball-likeness." Area still gates entry via `MIN_AREA = 20`, but is not a cost input. Weights are persisted in `data/candidate_selector_tuning.json` and tunable from the dashboard (defaults `w_aspect=0.6`, `w_fill=0.4`). The same selector runs in the live path via `live_pairing._resolve_candidates`; iOS ships aspect/fill on every candidate (lockstep wire contract). Historical pitch JSONs predating the aspect/fill wire fields reload with `aspect=fill=None` → both penalties evaluate to 0, costs all collapse, and argmin tie-break (first index) decides — not a useful selection on legacy data; new captures carry the stats and re-score honestly. **No `chain_filter` step** — the segmenter on `/fit/{sid}` supersedes its flicker/jump heuristics with a residual + running-RMSE gate against the multi-segment ballistic LSQ.
- `video.py` — PyAV decoder. `iter_frames(mov_path, video_start_pts_s) → Iterator[(absolute_pts_s, bgr)]`. Reconstructs absolute session-clock PTS: `absolute_pts_s = video_start_pts_s + (frame.pts − first_pts) * time_base`, so every decoded frame's timestamp sits on the same clock as `sync_anchor_timestamp_s`. `count_frames(path)` is a cheap second-pass counter used by tests.
- `pipeline.py` — glue. `detect_pitch(mov_path, video_start_pts_s, hsv_range?, frame_iter?, *, should_cancel?, shape_gate?, selector_tuning?, progress?)` decodes the MOV and runs the byte-aligned HSV → CC → shape gate → shape-prior selector pipeline (no MOG2, no morphology, no `expected_radius_px`, no chain_filter, no temporal-prior plumbing). Synthesises one `FramePayload` per decoded sample on the iOS session clock. Tests inject a stub iterator to avoid real decoding.
- `pairing.py` — `triangulate_cycle(a, b)` does A/B frame pairing within an 8 ms window of anchor-relative time (`timestamp_s − sync_anchor_timestamp_s`) and runs ray-midpoint triangulation. Each matched frame iterates `cands_a × cands_b` (cartesian fan-out); a pair is emitted iff both candidates pass `tuning.cost_threshold` AND the resulting skew-line gap ≤ `tuning.gap_threshold_m`. **`gap_threshold_m` is the sole authority for residual culling** — segmenter and viewer trust pairing's emit set, no downstream re-filter (Phase 3 of the residual-as-cost-sibling refactor; before that, `segmenter.find_segments` re-applied a hardcoded 0.20m gate). `_ray_for_frame` prefers the pixel path whenever `px`/`py` are present (server detection always produces them); zero-distortion is the default when `intrinsics.distortion` is absent, which is numerically equivalent to the legacy angle ray. Requires intrinsics + homography on both cameras.
- `triangulate.py` — `recover_extrinsics` decomposes `H = K [r1 r2 t]` (Zhang's planar method), orthonormalizes via SVD, flips sign if `t[2] < 0`. `triangulate_rays` solves the 2×2 system for the shortest segment between two 3D rays. `undistorted_ray_cam` applies OpenCV 5-coefficient undistortion.
- `reconstruct.py` — pure-geometry scene builder used by the viewer endpoints. `build_scene(session_id, pitches, triangulated)` returns a `Scene` with cameras, rays (tagged with `source` = `live` / `server` so the viewer can split them), ground traces per path (`ground_traces`, `ground_traces_live`), and triangulated points per path (`triangulated`). Viewer JS keeps the two pipelines independent via these per-path collections; they are never merged. `build_calibration_scene(calibrations)` builds the dashboard-canvas scene without rays. Both paths share `_camera_view_from_intrinsics_and_homography` so poses are byte-for-byte identical across `CalibrationSnapshot` and post-triangulation `PitchPayload`. **`t_rel_s` on rays and ground traces MUST share the same clock as `frames[*].t_rel_s` produced by `routes/viewer.py::_videos_for_session`** — both fall back to `video_start_pts_s` when `sync_anchor_timestamp_s` is None. A stale `or 0.0` fallback here would emit absolute PTS (~14681 s) while frames and the video element stay at anchor-relative (0–5 s), breaking viewer playback's scrubber→ray lookup.
- `chirp.py` — `chirp_wav_bytes()` builds the reference sync chirp WAV (2→8 kHz linear sweep, 100 ms, Hann-windowed, surrounded by 0.5 s silence) and is `@functools.lru_cache(maxsize=1)`'d so `GET /chirp.wav` reuses the exact bytes across requests.
- `render_scene.py` / `render_dashboard_page.py` (+ split helpers `render_dashboard_{client,devices,events,intrinsics,session,style}.py`) — Plotly viewer + dashboard HTML. Same PHYSICS_LAB palette. Plotly.js loaded once from CDN; SSR + JS ticks handle hydration. The earlier `render_dashboard.py` re-export facade was removed; callers import directly from the split modules.
- `main.py` — FastAPI app + `State` class (thread-locked). WS handler `ws_device` lives in `routes/device_ws.py`. `State.pitches` keyed by `(camera_id, session_id)`; when both A and B for a session arrive, per-path triangulation runs (see `session_results.py`) and the `SessionResult` is cached in `State.results[session_id]`. The `/pitch` handler lives in `routes/pitch.py`: multipart `payload` is always required, `video` arrives on every real upload (iOS records unconditionally post-PR61) but the handler accepts frame-only payloads so tests stay slim. `/pitch` no longer kicks off server-side detection; it just archives the MOV. Operator-triggered `POST /sessions/{sid}/run_server_post` schedules HSV detection as a FastAPI BackgroundTask; the candidate gate is "MOV on disk + session not trashed" (re-runs on already-completed sessions are allowed — the viewer's Rerun button relies on this). The request body **requires** a `source` field — `live` (current dashboard config), `frozen` (per-pitch `*_used` snapshot, 409 if any cam in the session lacks one), or `preset:<name>` (canonical HSV from `presets.PRESETS` without mutating disk — the research-compare path; shape_gate/selector still come from current state since presets are HSV-only). Both HTML form callers (events row "Run srv" button hard-pins `source=live`; viewer page exposes a `<select>` for all three sources) submit explicitly; per CLAUDE.md no-silent-fallback, an empty body is rejected (400). The detection-config triple (HSVRange / ShapeGate / CandidateSelectorTuning) is resolved **before** `resume_processing` mutates job state — a frozen-snapshot-missing failure leaves no zombie "queued" chips behind. The triple is then threaded into the BackgroundTask, so a `preset:blue_ball` rerun cannot be poisoned by a concurrent dashboard slider edit. Live `frame` / `path_completed` messages on `WS /ws/device/{cam}` go through `state.ingest_live_frame` + `state.live_rays_for_frame` (one ray per shape-gate-passing candidate, fan-out parity with the post-pitch viewer scene); `state.persist_live_frames` flushes the WS buffer onto the pitch JSON so a restart can replay. Other endpoints live under `routes/{sessions,calibration,markers,settings,sync,camera,viewer,pitch,fit}.py` — grep `@router\.(get|post|delete)` there for the canonical list. Notable groups: `/detection/{hsv,shape_gate,candidate_selector}` (operator-tunable detection knobs, persisted under `data/`), `/settings/{chirp_threshold,mutual_sync_threshold,heartbeat_interval,tracking_exposure_cap,capture_height,sync_params}`, `/sessions/{sid}/run_server_post` + `/sessions/{sid}/cancel_processing` (operator-triggered server-post detection + Cancel button on the events row), `/fit/{sid}` (multi-segment ballistic fit page), `/sync/*` (mutual sync coordinator), `/calibration/intrinsics/*` (per-device ChArUco upload). Removed: `/sessions/set_mode`, `/detection/paths` (legacy path-selection UI gone), `/detection/chain_filter` (chain_filter stack removed; segmenter supersedes).
- `state_events.py` / `session_results.py` / `detection_paths.py` (extracted from `state.py` in commit `60c3a95`) — `state_events.py` builds the `/events` dashboard rows (per-session, per-pipeline counts + path status pills); `session_results.py::rebuild_result_for_session` is the authoritative constructor for `SessionResult`, coordinating per-path triangulation; `detection_paths.py` resolves which frame bucket (`frames_live` / `frames_server_post`) is authoritative for a given `(pitch, path)` pair. All three are pure read against `State` — no mutation.
- `routes/device_ws.py` — PR #85 phase 2 abstraction: contains the `ws_device` handler that processes `hello` / `heartbeat` / `frame` / `cycle_end` WS messages on `/ws/device/{cam}`. Pulled out of `main.py` so the FastAPI app file stays the wiring root.
- `routes/calibration.py` — `POST /calibration` (manual `CalibrationSnapshot` upload), `GET /calibration/{cam}`, plus the homography / extrinsics dashboard endpoints. Owns the `_IPHONE_MAIN_CAM_HFOV_RAD = 1.2885` fallback used when iOS does not (yet) ship intrinsics.
- `routes/calibration_intrinsics.py` — split out in PR #93: per-device `/calibration/intrinsics/*` ChArUco intrinsics flow (upload, store, recompute) so the manual snapshot / homography / intrinsics surfaces stay unentangled. Auto-cal reruns preserve existing intrinsics and only update homography (Phase-3 contract).
- `live_pairing.py` — `LivePairingSession` buffers WS-streamed frame messages and pairs A/B within the 8 ms window on the fly. iOS ships every shape-gate-passing candidate per frame as `{px, py, area, area_score, aspect, fill}` (the on-wire candidate shape post-abfa422; iOS hand-encodes via `LiveFrameDispatcher.swift:93-95` so aspect/fill must be added explicitly there alongside the Codable struct). `area_score` is **wire-carried but not a selector-cost input** — kept only as a viewer BLOBS-overlay sort fallback (see `BlobCandidate` docstring in `schemas.py`). `cost` is **not** sent by iOS; the server stamps it onto every candidate inside `_resolve_candidates` after running the shape-prior selector, so persisted JSONs and downstream consumers see a populated `cost` field. `_resolve_candidates` runs the shared shape-prior `candidate_selector` (frame-local — no temporal state, no `area_score` term in the cost) so each cam's chosen blob is consistent with the server-side knobs in `data/candidate_selector_tuning.json`. The selector cost is `w_aspect·aspect_pen + w_fill·fill_pen` only; absolute `area` still gates entry via `MIN_AREA = 20` but is not a cost input (a 240 fps pitch sweeps r ≈ 4 → 25 px so absolute size cannot be a property of "ball-likeness"). The session's `tuning` field is hot-swapped by `state.ingest_live_frame` so dashboard slider updates apply on the next frame, no re-arm. Triangulated points are emitted as soon as the second cam's matching frame lands; the server broadcasts `path_completed` / triangulated-point events on the same WS channel so the dashboard can draw before the session ends.

For the `State` lock model + control plane (device registry / session / command
dispatch), see [architecture.md](architecture.md#dashboard-control-plane).
