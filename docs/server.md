# Server

Source under `server/`. FastAPI + `python-multipart` + PyAV + OpenCV.
`uv`-managed venv at `server/.venv` (Python 3.13).

## Key Python modules

- `schemas.py` — Pydantic types (`IntrinsicsPayload`, `BlobCandidate`, `FramePayload`, `PitchPayload`, `CalibrationSnapshot`, `TriangulatedPoint`, `SessionResult`, `DeviceIntrinsics`, `MarkerRecord` + draft/batch variants, `SyncReport` / `SyncResult` / `SyncRun` / `SyncLogEntry`) plus the `DetectionPath` / `TrackingExposureCapMode` enums and in-memory `Device` / `Session` / `StoredPitch` dataclasses. `BallisticSummary` and `CaptureMode` are gone (RANSAC fit removed). `SegmentRecord` carries the persisted shape of `segmenter.Segment` — `(p0, v0, t_anchor, t_start, t_end, indices, original_indices, rmse_m, speed_kph)` — and `SessionResult.segments: list[SegmentRecord]` is the single source of truth for multi-segment ballistic fit. The segmenter runs at result build / recompute time inside `session_results.stamp_segments_on_result` (which also pre-sorts `triangulated` by `t_rel_s` so `original_indices` indexes into a time-sorted list); the dashboard + viewer both read `result.segments` directly. Sample curves are NOT persisted — clients reconstruct via `p0 + v0·τ + ½·G·τ²`. Migration: legacy result JSONs that carried `ballistic_*` / `peak_z_m` / `ballistic_speed_mph` need a one-shot rewrite via `migrate_results_segments.py` (drops legacy keys + re-stamps segments). The standalone `/fit/{sid}` page is retired. `PitchPayload` carries two parallel frame buckets — `frames_live`, `frames_server_post` — each independently populated by the matching path; `PitchPayload.server_post_ran_at` (optional unix timestamp) is stamped at completion of a successful detection run for the viewer's "last run" stamp. `FramePayload.candidates: list[BlobCandidate]` is the Phase B wire shape: iOS ships every HSV-and-shape-gate-passing blob per frame (no top-K cap, no winner pre-selection); the server fans them all out through `pairing.triangulate_cycle`, which emits every geometrically-plausible pair (gated only by the absolute emit ceiling — see `pairing.py` entry above). `PairingTuning.gap_threshold_m` / `cost_threshold` are operator-stamped per-session filters applied downstream by the segmenter and the viewer slider; they do NOT gate emit. The viewer renders the full emitted set and the slider masks client-side, so dragging Cost ≤ / Gap ≤ is purely a render-time mask; Apply re-runs the segmenter on the stamped subset. `BlobCandidate` carries `(px, py, area, area_score, aspect, fill, cost)` — `aspect`/`fill` are server-populated by detection.py and consumed by the shape-prior cost stamped onto each candidate so the viewer's Cost slider (sibling of the Gap slider in the per-session tuning strip) can filter cheap distractors. `SessionResult.cost_threshold` and `SessionResult.gap_threshold_m` persist the operator's chosen per-session values; both are sent in the body of `POST /sessions/{sid}/recompute` and re-stamped on the resulting result so the viewer header re-initialises after reload. `recompute_result_for_session` re-pairs from candidates (under the absolute emit ceiling, ignoring the stamped values) and the segmenter then runs against the stamped-tuning SUBSET — so Apply changes which points are FIT without changing which points are PERSISTED. `TriangulatedPoint.cost_a` / `cost_b` (source candidates' costs) ride along on every emitted point so the client-side cost filter (`_passCostFilterPoint` in `static/viewer/50_canvas.js`) can reproduce the same predicate without a (cam, frame, cand_idx) lookup. `PitchPayload.{hsv_range_used, shape_gate_used}` and the matching `SessionResult.*_used` mirrors freeze the **detection-time config** at first persist of each pitch — `routes/pitch.py` stamps from the live session's first-ingest snapshot on /pitch upload and re-stamps from the actual server_post call args on completion; `state.ingest_live_frame` freezes `live.pairing_tuning` / `live.hsv_range_used` / `live.shape_gate_used` on first ingest only (dashboard slider edits during an active session land on the next session). Selector cost weights are no longer in this snapshot — they're locked at `_W_ASPECT=0.6` / `_W_FILL=0.4` module constants in `candidate_selector.py`. `reprocess_sessions.py` defaults to current `data/*.json` so the operator's tuning workflow (edit slider → rerun → see if it improves) actually picks up changes; opt-in `--use-frozen-snapshot` reuses `pitch.*_used` for reproducibility audits and falls back to disk on legacy pitches with a logged warning per missing field. The frozen stamps on disk are always re-written at end of run with whatever config the run actually used, so the "what config produced this pitch" question stays answerable even after a default-mode rerun overwrites them. Same intent as cd87995's PairingTuning-on-SessionResult fix, generalised to detection. `CalibrationSnapshot` is the body of `POST /calibration`. `camera_id` is `^[A-Za-z0-9_-]{1,16}$`; `session_id` regex is `^s_[0-9a-f]{4,32}$` (server mints `s_` + 8 hex chars via `secrets.token_hex(4)`; 32-char upper bound is forward-compat slack). Both path-safe.
- `detection.py` — OpenCV-backed HSV blob extractor + multi-candidate scorer. `HSVRange.default()` is `25,55,90,255,90,255` (fluorescent yellow-green tennis ball — fallback for headless tests / first boot); the dashboard persists the live config (HSV + shape_gate pair) atomically in `data/detection_config.json`. The unified card POSTs the pair to `POST /detection/config` (single Apply button, optional `preset` identity claim that must match the values exactly) and `POST /detection/config/reset_to_preset` snaps back to a named preset. The legacy per-section endpoints (`/detection/hsv`, `/detection/shape_gate`, `/detection/candidate_selector`) are retired — calls return 404. On boot, `detection_config.load_or_migrate` migrates legacy `data/{hsv_range,shape_gate,candidate_selector_tuning}.json` into the unified file once and deletes them (the selector legacy file is unlinked without parsing — its values are no longer loadable). Subsequent boots read only the new file. The named preset library is **disk-backed**: `presets.py` exposes `load_preset` / `list_presets` / `save_preset` / `delete_preset` over `data/presets/<name>.json`, with `seed_builtins` writing the `tennis` and `blue_ball` seeds on first boot if their files do not exist. Built-ins survive operator edits (the seed step never overwrites an existing file) and an `rm` + restart restores the canonical seed. Operator-facing CRUD lives under `GET /presets`, `GET /presets/{name}`, `POST /presets` (409 on duplicate), `PUT /presets/{name}` (404 if missing — URL is canonical for the slug), `DELETE /presets/{name}`. Slug is restricted to `[a-z0-9_]{1,32}` so it doubles as a filename safely; the operator-facing `label` is free-form and HTML-escaped at render. Identity-claim validation on `POST /detection/config` reads from disk, so a freshly-saved preset becomes claimable as `preset=<name>` immediately without restart. The `presets.PRESETS` module-level dict is gone. `HSVRange.from_env()` still reads `BALL_TRACKER_HSV_RANGE="hMin,hMax,sMin,sMax,vMin,vMax"` as a last-resort override for headless test runs. Hue is OpenCV's 8-bit convention (0-179, standard 0-360° halved). `detect_ball(bgr, hsv_range, ...)` runs `cv2.inRange` + `cv2.connectedComponentsWithStats`, applies the operator-tunable `ShapeGate` (aspect / fill thresholds, persisted inside the unified detection_config), and feeds the surviving blobs to `candidate_selector.score_candidates`. The selector cost is **track-independent and scale-invariant**: `cost = _W_ASPECT·aspect_pen + _W_FILL·fill_pen` where `aspect_pen` is `(1 - aspect)` normalised to a fixed floor (0.5), and `fill_pen` is `|fill - 0.68|/0.68` (0.68 = empirical median for the project ball). There is no size term — absolute area in pixels is a function of distance (a 240 fps pitch sweeps r ≈ 4 → 25 px), so it cannot be a property of "ball-likeness." Area still gates entry via `MIN_AREA = 20`, but is not a cost input. Weights are locked at `_W_ASPECT=0.6 / _W_FILL=0.4` module constants in `candidate_selector.py` — there is no runtime tunable, no disk file, no dashboard card. The same selector runs in the live path via `live_pairing._resolve_candidates`; iOS ships aspect/fill on every candidate (lockstep wire contract). Historical pitch JSONs predating the aspect/fill wire fields reload with `aspect=fill=None` → both penalties evaluate to 0, costs all collapse, and argmin tie-break (first index) decides — not a useful selection on legacy data; new captures carry the stats and re-score honestly. **No `chain_filter` step** — `segmenter.find_segments` (run at SessionResult build time, persisted on `SessionResult.segments`) supersedes its flicker/jump heuristics with a residual + running-RMSE gate against the multi-segment ballistic LSQ.
- `video.py` — PyAV decoder. `iter_frames(mov_path, video_start_pts_s) → Iterator[(absolute_pts_s, bgr)]`. Reconstructs absolute session-clock PTS: `absolute_pts_s = video_start_pts_s + (frame.pts − first_pts) * time_base`, so every decoded frame's timestamp sits on the same clock as `sync_anchor_timestamp_s`. `count_frames(path)` is a cheap second-pass counter used by tests.
- `pipeline.py` — glue. `detect_pitch(mov_path, video_start_pts_s, hsv_range?, frame_iter?, *, should_cancel?, shape_gate?, progress?)` decodes the MOV and runs the byte-aligned HSV → CC → shape gate → shape-prior selector pipeline (no MOG2, no morphology, no `expected_radius_px`, no chain_filter, no temporal-prior plumbing). Synthesises one `FramePayload` per decoded sample on the iOS session clock. Tests inject a stub iterator to avoid real decoding.
- `pairing.py` — `triangulate_cycle(a, b)` does A/B frame pairing within an 8 ms window of anchor-relative time (`timestamp_s − sync_anchor_timestamp_s`) and runs ray-midpoint triangulation. Each matched frame iterates `cands_a × cands_b` (cartesian fan-out). **Pairing emits the FULL set** of geometrically-plausible candidate pairs, gated only by absolute module-level ceilings `_EMIT_COST_CEILING=5.0` / `_EMIT_GAP_CEILING_M=5.0` — disk/memory protection, NOT operator-tunable. `tuning.cost_threshold` / `tuning.gap_threshold_m` are the OPERATOR per-session filters that the segmenter (`session_results.stamp_segments_on_result`) and the viewer's slider (`static/viewer/50_canvas.js::_passCostFilterPoint` + `_passResidualFilter`) apply DOWNSTREAM; they no longer gate emit. Every emitted `TriangulatedPoint` carries `cost_a` / `cost_b` (source candidates' selector cost) so the client cost filter has the data; `triangulated_by_path` on disk is the full emitted set so reload + Apply can re-fit any subset without re-pairing. `_ray_for_frame` prefers the pixel path whenever `px`/`py` are present (server detection always produces them); zero-distortion is the default when `intrinsics.distortion` is absent, which is numerically equivalent to the legacy angle ray. Requires intrinsics + homography on both cameras.
- `triangulate.py` — `recover_extrinsics` decomposes `H = K [r1 r2 t]` (Zhang's planar method), orthonormalizes via SVD, flips sign if `t[2] < 0`. `triangulate_rays` solves the 2×2 system for the shortest segment between two 3D rays. `undistorted_ray_cam` applies OpenCV 5-coefficient undistortion.
- `reconstruct.py` — pure-geometry scene builder used by the viewer endpoints. `build_scene(session_id, pitches, triangulated)` returns a `Scene` with cameras, rays (tagged with `source` = `live` / `server` so the viewer can split them), ground traces per path (`ground_traces`, `ground_traces_live`), and triangulated points per path (`triangulated`). Viewer JS keeps the two pipelines independent via these per-path collections; they are never merged. `build_calibration_scene(calibrations)` builds the dashboard-canvas scene without rays. Both paths share `_camera_view_from_intrinsics_and_homography` so poses are byte-for-byte identical across `CalibrationSnapshot` and post-triangulation `PitchPayload`. **`t_rel_s` on rays and ground traces MUST share the same clock as `frames[*].t_rel_s` produced by `routes/viewer.py::_videos_for_session`** — both fall back to `video_start_pts_s` when `sync_anchor_timestamp_s` is None. A stale `or 0.0` fallback here would emit absolute PTS (~14681 s) while frames and the video element stay at anchor-relative (0–5 s), breaking viewer playback's scrubber→ray lookup.
- `chirp.py` — `chirp_wav_bytes()` builds the reference sync chirp WAV (2→8 kHz linear sweep, 100 ms, Hann-windowed, surrounded by 0.5 s silence) and is `@functools.lru_cache(maxsize=1)`'d so `GET /chirp.wav` reuses the exact bytes across requests.
- `scene_runtime.py` / `render_dashboard_page.py` / `viewer_page.py` (+ split helpers `render_dashboard_{client,devices,events,intrinsics,session,style}.py`) — Three.js dashboard + viewer HTML. Same PHYSICS_LAB palette. The shared 3D scene runtime lives client-side at `static/threejs/scene_runtime.js` (renderer + camera + OrbitControls + the static layers ground / plate / strike zone / world axes); per-page layer modules in `static/threejs/{dashboard,viewer}_layers.js` add dynamic content. `scene_runtime.py::scene_theme()` ships the constants both consume as a JSON-safe payload. Three.js is self-hosted in `static/threejs/vendor/`; `main.py` mounts `/static/` via `StaticFiles` and fails loud at boot if the vendored files are missing (`vendor_files_present`).
- `main.py` — FastAPI app + `State` class (thread-locked). WS handler `ws_device` lives in `routes/device_ws.py`. `State.pitches` keyed by `(camera_id, session_id)`; when both A and B for a session arrive, per-path triangulation runs (see `session_results.py`) and the `SessionResult` is cached in `State.results[session_id]`. The `/pitch` handler lives in `routes/pitch.py`: multipart `payload` is always required, `video` arrives on every real upload (iOS records unconditionally post-PR61) but the handler accepts frame-only payloads so tests stay slim. `/pitch` no longer kicks off server-side detection; it just archives the MOV. Operator-triggered `POST /sessions/{sid}/run_server_post` schedules HSV detection as a FastAPI BackgroundTask; the candidate gate is "MOV on disk + session not trashed" (re-runs on already-completed sessions are allowed — the viewer's Rerun button relies on this). The request body **requires** a `source` field — `live` (current dashboard config), `frozen` (per-pitch `*_used` snapshot, 409 if any cam in the session lacks one), or `preset:<name>` (canonical (HSV, shape_gate) pair loaded from `data/presets/<name>.json` without mutating the live `detection_config.json` — the research-compare path). Both HTML form callers (events row "Run srv" button hard-pins `source=live`; viewer page exposes a `<select>` for all three sources) submit explicitly; per CLAUDE.md no-silent-fallback, an empty body is rejected (400). The detection-config pair (HSVRange / ShapeGate) is resolved **before** `resume_processing` mutates job state — a frozen-snapshot-missing failure leaves no zombie "queued" chips behind. The pair is then threaded into the BackgroundTask, so a `preset:blue_ball` rerun cannot be poisoned by a concurrent dashboard slider edit. Live `frame` / `path_completed` messages on `WS /ws/device/{cam}` go through `state.ingest_live_frame` + `state.live_rays_for_frame` (one ray per shape-gate-passing candidate, fan-out parity with the post-pitch viewer scene); `state.persist_live_frames` flushes the WS buffer onto the pitch JSON so a restart can replay. Other endpoints live under `routes/{sessions,calibration,markers,settings,sync,camera,viewer,pitch}.py` — grep `@router\.(get|post|delete)` there for the canonical list. Notable groups: `/detection/config` + `/detection/config/reset_to_preset` (the unified detection-config pair endpoints — the legacy per-section `/detection/{hsv,shape_gate,candidate_selector}` endpoints are retired and return 404), `/settings/{chirp_threshold,mutual_sync_threshold,heartbeat_interval,tracking_exposure_cap,capture_height,sync_params}`, `/sessions/{sid}/run_server_post` + `/sessions/{sid}/cancel_processing` (operator-triggered server-post detection + Cancel button on the events row), `/sync/*` (mutual sync coordinator), `/calibration/intrinsics/*` (per-device ChArUco upload). The standalone `/fit/{sid}` page is retired (`routes/fit.py` + `render_fit.py` deleted; `SessionResult.segments` is now the canonical surface). Removed: `/sessions/set_mode`, `/detection/paths` (legacy path-selection UI gone), `/detection/chain_filter` (chain_filter stack removed; segmenter supersedes).
- `state_events.py` / `session_results.py` / `detection_paths.py` (extracted from `state.py` in commit `60c3a95`) — `state_events.py` builds the `/events` dashboard rows (per-session, per-pipeline counts + path status pills); `session_results.py::rebuild_result_for_session` is the authoritative constructor for `SessionResult`, coordinating per-path triangulation; `detection_paths.py` resolves which frame bucket (`frames_live` / `frames_server_post`) is authoritative for a given `(pitch, path)` pair. All three are pure read against `State` — no mutation.
- `routes/device_ws.py` — PR #85 phase 2 abstraction: contains the `ws_device` handler that processes `hello` / `heartbeat` / `frame` / `cycle_end` WS messages on `/ws/device/{cam}`. Pulled out of `main.py` so the FastAPI app file stays the wiring root.
- `routes/calibration.py` — `POST /calibration` (manual `CalibrationSnapshot` upload), `GET /calibration/{cam}`, plus the homography / extrinsics dashboard endpoints. Owns the `_IPHONE_MAIN_CAM_HFOV_RAD = 1.2885` fallback used when iOS does not (yet) ship intrinsics.
- `routes/calibration_intrinsics.py` — split out in PR #93: per-device `/calibration/intrinsics/*` ChArUco intrinsics flow (upload, store, recompute) so the manual snapshot / homography / intrinsics surfaces stay unentangled. Auto-cal reruns preserve existing intrinsics and only update homography (Phase-3 contract).
- `calibration_auto.py` — pure auto-calibration business logic extracted from `routes/calibration.py` (PR `a9d46f6`, brought `routes/calibration.py` down from 1162 → 298 lines). Owns the JPEG → 4:3-to-16:9 center-crop → ArUco detect → intrinsics derivation (ChArUco prior / cached snapshot / `_IPHONE_MAIN_CAM_HFOV_RAD = 1.2885` fallback at `calibration_auto.py:51`) → planar homography → PnP refinement → reprojection gate → rebuild in canonical 1920×1080 video basis. `_run_auto_calibration` is the single entrypoint; FastAPI route handlers in `routes/calibration.py` are thin wrappers. Module raises `HTTPException` directly so callers don't need try/except wrapping.
- `live_pairing.py` — `LivePairingSession` buffers WS-streamed frame messages and pairs A/B within the 8 ms window on the fly. iOS ships every shape-gate-passing candidate per frame as `{px, py, area, area_score, aspect, fill}` (the on-wire candidate shape post-abfa422; iOS hand-encodes via `LiveFrameDispatcher.swift:93-95` so aspect/fill must be added explicitly there alongside the Codable struct). `area_score` is **wire-carried but not a selector-cost input** — kept only as a viewer BLOBS-overlay sort fallback (see `BlobCandidate` docstring in `schemas.py`). `cost` is **not** sent by iOS; the server stamps it onto every candidate inside `_resolve_candidates` after running the shape-prior selector, so persisted JSONs and downstream consumers see a populated `cost` field. `_resolve_candidates` runs the shared shape-prior `candidate_selector` (frame-local — no temporal state, no `area_score` term in the cost). The selector cost is `_W_ASPECT·aspect_pen + _W_FILL·fill_pen` with weights locked as module constants (no runtime tunable, no disk file); absolute `area` still gates entry via `MIN_AREA = 20` but is not a cost input (a 240 fps pitch sweeps r ≈ 4 → 25 px so absolute size cannot be a property of "ball-likeness"). Triangulated points are emitted as soon as the second cam's matching frame lands; the server broadcasts `path_completed` / triangulated-point events on the same WS channel so the dashboard can draw before the session ends.

For the `State` lock model + control plane (device registry / session / command
