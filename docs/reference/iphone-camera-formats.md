# iPhone rear main (wide) camera format dump

Generated by the debug logging added in PR #66 (`ball_tracker` at startup).
Device: `Back Camera` (`builtInWideAngleCamera`), uniqueID
`com.apple.avfoundation.avcapturedevice.built-in_video:0`.

> Record the device model alongside any future dump — FOV and format
> availability drift between iPhone generations.

## Key findings

### 240 fps is gated to two binned formats

Only `format[22]` and `format[39]` support 240 fps, and **both are binned**
(sensor pixel binning active):

| idx | dims       | max fps | binned | hFOV° | subtype  |
|-----|------------|---------|--------|-------|----------|
| 22  | 1280x720   | 240     | yes    | 73.83 | 34323076 |
| 23  | 1280x720   | 240     | yes    | 73.83 | 34323066 |
| 38  | 1920x1080  | 240     | yes    | 73.83 | 34323076 |
| 39  | 1920x1080  | 240     | yes    | 73.83 | 34323066 |

`format[38]`/`format[39]` is the production tracking path (1920x1080 @ 240 fps).
Binning means the sensor reads 2×2 pixel groups as one sample — lower
effective resolution but dramatically faster readout, which is why 240 fps
is only reachable here.

### FOV is 73.83°

✅ **Shipped.** The server's fallback FOV prior is now `1.2885 rad ≈ 73.828°`
(`server/calibration_auto.py:51` — `_IPHONE_MAIN_CAM_HFOV_RAD`). Historical context
preserved below for git-blame archaeology.

The dump says 73.828° for every 1920x1080 16:9 format except a 120 fps
telephoto-crop variant at 41.173°. Earlier code defaulted to 65°, which
derives `fx ≈ 1507` for 1920px instead of the correct `fx ≈ 1278` —
a ~14% over-estimate of `fx`, enough to push recovered depth 10–15%
closer than reality and explain a chunk of the sub-metre pose
discrepancies we used to chase. Both the server prior and any iOS
persisted FOV default (`horizontalFovRadians`) are now aligned to the
73.828° value. The 41.173° telephoto-crop format is unused; ultra-wide
is rejected.

### Alternative framings available (not used)

- `format[41–46]` — 1920x1440 4:3 @ up to 60 fps, FOV 71.286°. Wider
  vertical view; no 240 fps. Not worth the FPS loss.
- `format[36–37]` — 1920x1080 @ 120 fps with FOV 41.173° (cropped-sensor
  telephoto readout). Useful if we ever want a narrow-field capture, but
  FOV mismatch would break pairing with a 73.8° cam.
- `format[51–57]` — 3840x2160 4K up to 60 fps, FOV 73.828°. No 240 fps.
  Could be a "slow-motion-off" preview path but uploads would quadruple.
